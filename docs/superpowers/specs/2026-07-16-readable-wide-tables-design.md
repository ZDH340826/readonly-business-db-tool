# 多列表格横向浏览设计

## 目标

当表格列数较多或内容较长时，不再把列名和内容强行压缩到当前框宽。用户应能先看清字段含义，再通过横向滚动查看其余列；软件仍保持只读，数据查询和业务逻辑不变。

## 适用范围

- 统一应用于软件内所有使用标准表格样式的数据表格。
- 截图中的“对象元数据与列信息”和“前 100 行只读预览”必须生效。
- 总列宽未超过可视区域时，不显示多余的横向滚动条。
- 树、表单、文本详情和导航栏不受影响。

## 交互设计

1. 列名必须按可读宽度显示，不因窗口较窄而被压成难以理解的省略文本。
2. 列宽根据表头和当前数据计算，保留必要留白，并设置合理的最小值与最大值，避免过窄或单列无限扩张。
3. 总列宽超过表格框时，底部按需出现横向滚动条；滚动时表头与数据同步移动。
4. 用户可拖动横向滚动条；系统支持时，可使用触摸板横向手势或 `Shift + 鼠标滚轮`。
5. 数据重新加载、表结构变化或预览列变化后，重新计算列宽；空表也应依据表头保持可读。

## 实现边界

- 在统一表格工厂中集中配置非强制压缩模式和列宽计算，页面不重复实现相同逻辑。
- 现有 `JScrollPane` 保持“需要时显示”横向滚动条。
- 不改变 SQL、数据库连接、只读限制、返回行数、排序、筛选、报警或监控逻辑。
- 不修改旧版页面结构，不进行无关界面美化。

## 异常与边界情况

- `null`、空字符串和空数据集不影响列宽计算。
- 超长单元格按最大列宽截断显示，避免一条异常数据制造极长滚动区；完整值仍可通过现有详情区域查看。
- 模型在后台加载数据时，列宽更新切回 Swing 界面线程执行，避免界面竞态。

## 验证标准

- 自动测试证明标准数据表不再使用强制压缩列模式。
- 自动测试证明表头和样例数据得到可读的首选宽度，模型结构变化后宽度会更新。
- 自动测试证明宽表的滚动范围大于可视范围，窄表不产生无意义的横向滚动范围。
- 使用本地 H2 示例数据库在 1440×900 界面打开数据源浏览器，人工确认红框内两张表可横向滚动、表头清楚、滚动时表头与数据同步。
- 完整构建、全部测试、部署预检和 `git diff --check` 通过。
