# v0.4.0 点位状态看板设计

## 目标

将点位缺料报警页从“配置表单 + 技术日志”升级为现场运维可直接使用的“点位状态看板”。

v0.4.0 的核心目标是：

```text
打开点位缺料报警页后，不看英文日志、不推理规则，直接看到每个点位是“有料”还是“无料”。
```

本版本继续保持 v0.3.0 的只读数据库边界，不增加任何数据库写入能力。

## v0.3.0 基线

v0.3.0 已具备：

- 现场 PostgreSQL 连接配置，默认适配 `cms_web.public`。
- 只读查询现场表 `public.tcs_map_data`。
- 本地保存连接配置，密码不保存。
- 点位组配置：区域、物料组、使用位、备用位。
- 点位组报警规则：使用位无料、备用位有料数量低于阈值、持续 N 分钟后报警。
- 检测日志和报警事件日志。
- 报警弹窗必须手动点击“已关注”。

v0.3.0 主要不足：

- 点位缺料报警页更像配置页，不像监控页。
- 点位是否有料不直观，需要从结果日志推断。
- 检测结果存在 `PENDING_ALERT useEmpty=true backup=0/6` 这类技术文本。
- 检测周期固定 60 秒，不适合不同消耗速度的物料。
- 持续报警时间和固定检测周期耦合，不利于后续扩展。
- 使用位没有被突出展示，现场重点不够清楚。

## v0.4.0 总览

v0.4.0 新增和优化：

- 点位卡片直接显示 `有料` / `无料` / `未查到` / `停用`。
- 点位组显示 `正常` / `观察中` / `需关注` / `已关注`。
- 每个点位组可单独配置检测周期。
- 每个点位组可单独配置报警持续时间。
- 报警判断改为基于真实经过时间，而不是简单按检测次数估算。
- 检测结果改为中文业务语言。
- 报警弹窗展示异常点位、持续时间和处理动作。

## 核心概念

| 概念 | 定义 |
|---|---|
| 检测周期 | 系统多久查询一次数据库。例如每 1 分钟检测一次。 |
| 报警持续时间 | 异常状态连续存在多久后报警。例如连续 5 分钟无料后报警。 |
| 连续异常开始时间 | 本轮异常第一次被检测到的时间。 |
| 使用位 | 当前消耗物料的点位，应优先关注。 |
| 备用位 | 用于暂存或递进补充使用位的点位。 |
| 有料 | 点位有可用货架：`pod_code` 非空、`status == 1`、`ind_lock == 0`。 |
| 无料 | 点位没有可用货架。 |
| 未查到 | 数据库没有返回该点位记录。 |
| 停用 | 本地配置了点位，但该点位未启用监控。 |

## 报警规则模型

每个点位组拥有独立的检测周期和报警规则。

必须支持的配置项：

- `检测周期`：默认 60 秒，可配置为 1 / 2 / 5 / 10 分钟或自定义。
- `报警持续时间`：默认 5 分钟，可自定义。
- `使用位连续无料报警`：默认启用。
- `备用位有料下限`：继续保留 v0.3.0 的备用位数量阈值。
- `备用位下限参与报警`：默认启用，用于兼容 v0.3.0；也允许关闭，让备用位数量只作为参考状态。

支持两种规则模式。

### 模式 A：只看使用位

适用于用户明确只关心“使用位是否被补料”的场景。

```text
条件：使用位无料
报警：使用位连续无料达到配置时间
```

示例：

```text
后围板组
检测周期 = 1 分钟
报警持续时间 = 5 分钟
规则 = 使用位连续无料 5 分钟后报警
```

### 模式 B：使用位 + 备用位阈值

适用于 v0.3.0 已有规则和更严格的缺料判断。

```text
条件：使用位无料，并且备用位有料数量低于下限
报警：上述条件连续达到配置时间
```

示例：

```text
后围板组
检测周期 = 1 分钟
报警持续时间 = 5 分钟
备用位有料下限 = 3
规则 = 使用位无料，并且备用位有料少于 3 个，持续 5 分钟后报警
```

## 时间判断逻辑

v0.4.0 不能再简单按检测次数判断持续时间，应使用真实时间。

每个点位组运行态需要记录：

- `lastCheckedAt`：最近检测时间。
- `conditionMatched`：当前异常条件是否成立。
- `conditionFirstMatchedAt`：本轮异常第一次成立时间。
- `conditionLastMatchedAt`：最近一次异常成立时间。
- `continuousSeconds`：连续异常秒数。
- `lastStatus`：最近一次组状态。
- `acknowledged`：本轮报警是否已关注。

判断逻辑：

```text
如果条件恢复正常：
    清空 conditionFirstMatchedAt
    清空 acknowledged
    状态 = 正常

如果条件异常：
    如果 conditionFirstMatchedAt 为空：
        conditionFirstMatchedAt = 当前时间

    continuousSeconds = 当前时间 - conditionFirstMatchedAt

    如果 continuousSeconds < 报警持续时间：
        状态 = 观察中
    否则如果本轮已关注：
        状态 = 已关注
    否则：
        状态 = 需关注，并弹窗
```

这样即使不同点位组使用 1 分钟、2 分钟或 10 分钟检测周期，报警持续时间也能保持准确。

## 点位状态展示

每个点位必须直接展示一个状态。

| UI 状态 | 判断规则 |
|---|---|
| 有料 | 返回记录存在，`pod_code` 非空，`status == 1`，`ind_lock == 0` |
| 无料 | 返回记录存在，但不满足有料规则 |
| 未查到 | 查询结果中没有该点位编码 |
| 停用 | 本地点位配置未启用 |

点位卡片展示字段：

- 点位角色：使用位 / 备用位。
- 点位别名。
- 点位编码。
- 状态：有料 / 无料 / 未查到 / 停用。
- 货架编号，有则显示。
- 更新时间，有则显示。
- 异常原因，例如 `无货架`、`锁定`、`点位未返回`、`已停用`。

使用位必须比备用位更醒目，可以使用更大的卡片或固定放在首位。

## 点位缺料报警页 UI

页面布局建议如下。

### 左侧导航

继续保留：

- `连接管理`
- `数据库浏览器`
- `点位缺料报警`

### 点位组列表

每个点位组行显示：

- 区域和组名。
- 状态徽标：正常 / 观察中 / 需关注 / 已关注。
- 使用位状态。
- 备用位有料数量，例如 `备用位 2/4 有料`。
- 最近检测时间。

### 当前点位组摘要

选中点位组后，顶部摘要显示：

- 区域。
- 组名。
- 物料。
- 当前状态。
- 使用位状态。
- 备用位有料数量。
- 连续异常时间。
- 检测周期。
- 报警持续时间。

### 点位状态看板

主体区域显示：

- 使用位大卡片。
- 备用位网格卡片。
- 每张卡片直接显示“有料 / 无料 / 未查到 / 停用”。

颜色约定：

- 绿色：有料 / 正常。
- 红色：无料 / 需关注。
- 黄色：观察中。
- 灰色：未查到 / 停用。

状态不能只靠颜色表达，必须有文字。

### 规则面板

规则面板显示自然语言说明：

```text
每 1 分钟检测一次。
使用位连续无料 5 分钟后报警。
备用位有料少于 3 个时参与报警判断。
```

可编辑项：

- 检测周期。
- 报警持续时间。
- 备用位有料下限。
- 备用位下限是否参与报警。

### 事件区

底部只显示中文摘要，不再把技术日志作为主结果。

示例：

```text
06:21:08 观察中：使用位无料，已持续 3/5 分钟，备用位 2/4 有料。
06:23:08 需关注：后围板使用位连续 5 分钟无料。
06:24:10 已关注：现场运维已确认本轮报警。
```

原始日志继续写入文件，供排查使用。

## 中文结果文本

主界面必须使用中文业务语言。

示例：

```text
正常：使用位有料，备用位 4/4 有料。
观察中：使用位无料，已持续 3/5 分钟，备用位 2/4 有料。
需关注：后围板使用位连续 5 分钟无料，请现场确认。
已关注：报警已确认，等待点位恢复正常。
未查到：配置点位未从数据库返回，请检查点位编码或数据同步状态。
```

`PENDING_ALERT`、`ACTIVE_ALERT`、`useEmpty=true` 等技术字段不得作为主界面结果展示。

## 报警弹窗

报警弹窗目标是让现场人员立即知道该处理什么。

弹窗内容：

- 标题：`点位缺料报警`。
- 主提示：`{区域} / {组名} 需要关注`。
- 状态摘要：
  - `使用位：无料`
  - `备用位：2/4 有料`
  - `持续：5 分钟`
  - `规则：连续无料报警`
- 异常点位列表。
- 处理说明：
  - `使用位无料已达到报警时间，请现场确认补料或调度状态。`

按钮：

- 主按钮：`已关注`。
- 次按钮：`打开日志目录`。

点击 `已关注` 后，本轮异常不重复弹窗。点位组恢复正常后，如果后续再次异常，可以重新报警。

## 配置变更

`data/group-config.properties` 需要保存新增字段：

- 点位组检测周期。
- 报警持续时间。
- 备用位下限是否参与报警。

兼容要求：

- v0.3.0 旧配置必须能加载。
- 旧配置缺少检测周期时，默认 60 秒。
- 旧配置缺少报警持续时间时，使用原有 `durationMinutes`。
- 旧配置缺少备用位参与标志时，默认启用，保持 v0.3.0 行为。

配置文件不得保存数据库密码。

## 数据流

1. 调度器根据每个点位组的检测周期判断哪些组到期。
2. 仓储层只读查询到期点位组的启用点位。
3. 点位状态映射器将配置点位和查询结果合并为状态模型。
4. 点位组评估器计算：
   - 使用位状态。
   - 备用位有料数量。
   - 报警条件是否成立。
   - 连续异常时间。
   - 点位组状态。
5. UI 刷新点位组列表、摘要、点位卡片和事件区。
6. 日志系统写入检测快照和事件变化。

## 非目标

v0.4.0 不做：

- 数据库写入。
- SQL 编辑器。
- 表格编辑。
- 地图渲染。
- 拖拽式点位布局。
- 自动修改现场点位数据。
- 保存数据库密码。
- Web 服务部署。

## 验证要求

必须增加测试覆盖。

点位状态映射测试：

- 有可用货架 -> `有料`。
- 无货架或锁定 -> `无料`。
- 查询结果缺失 -> `未查到`。
- 本地禁用 -> `停用`。

点位组时间评估测试：

- 第一次异常时记录连续异常开始时间。
- 未达到报警持续时间时为 `观察中`。
- 达到报警持续时间时为 `需关注`。
- 点击已关注后为 `已关注`。
- 恢复正常后清空本轮异常状态。

配置兼容测试：

- v0.3.0 旧配置可加载。
- v0.4.0 新字段可保存和重新加载。

UI 测试：

- 监控页包含 `有料`、`无料`、`未查到`、`停用`。
- 主结果不显示 `PENDING_ALERT` 等技术文本。

发布验证：

- `.\build.ps1` 通过。
- 发布包 self-test 通过。
- 源码和发布包隐私扫描通过。
- 不提交真实 IP、密码、真实点位组和生产点位清单。

## 验收标准

v0.4.0 可验收条件：

1. 现场人员打开点位缺料报警页，可以直接看到每个点位是 `有料`、`无料`、`未查到` 或 `停用`。
2. 当前点位组清楚显示 `正常`、`观察中`、`需关注` 或 `已关注`。
3. 可以将后围板组设置为每 1 分钟检测一次。
4. 可以将后围板组设置为使用位连续无料 5 分钟后报警。
5. 系统在未达到持续时间前只显示观察中，不弹窗。
6. 系统在达到持续时间后弹出中文报警窗口。
7. 点击 `已关注` 后，本轮异常不重复弹窗。
8. 点位恢复正常后，后续再次异常可以重新报警。
9. 数据库访问仍严格只读。
