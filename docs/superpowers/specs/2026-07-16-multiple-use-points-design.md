# 多使用位点位组设计

## 目标

允许一个点位组配置一个或多个启用的使用位，适配存在两个上料口的现场。报警逻辑继续使用已经确认的“前者”：**任一启用使用位无料，即满足使用位异常条件**。

本次只扩展点位组配置与报警判断，不改变数据库只读边界，不引入新的报警模式，也不修改备用位不足、异常持续时间、确认和恢复重新计时等现有语义。

## 使用者与表达原则

使用者是非技术运维人员。界面、校验和报警内容必须直接使用中文业务词：

- `USE` 显示为“使用位”；
- `BACKUP` 显示为“备用位”；
- `map_data_code` 对外统一称为“地码”；
- “使用位条件”明确显示为“任一启用使用位无料”；
- 不向运维显示枚举名、内部字段名、Java 异常或栈信息。

## 已选方案

### 方案 A：兼容现有模型，解除单使用位限制（采用）

现有 `PointGroupDefinition` 已使用点位列表保存角色，持久化格式也允许出现多条 `USE`。只需把“恰好一个启用使用位”的领域校验改为“至少一个启用使用位”，并为多使用位报警语义补齐测试与中文状态表达。

优点：

- 旧配置无需迁移；
- 不增加现场可误配的选项；
- 数据查询、调度、日志和只读边界保持不变；
- 改动集中，容易回归验证。

### 方案 B：增加“任一无料 / 全部无料”模式开关（不采用）

该方案更灵活，但与用户已经确认的固定报警语义冲突，并增加运维理解和误配置成本。

### 方案 C：把多个使用位合并为一个虚拟使用位（不采用）

该方案会丢失具体哪个上料口异常的信息，状态看板和现场处置不直观。

## 领域模型与兼容性

### 点位组校验

启用点位组必须满足：

1. 至少一个启用的使用位；
2. 至少一个启用的备用位；
3. 同组启用点位的地码不得重复；
4. 点位 ID 不得重复；
5. 备用位最少有料数不得超过启用备用位数量。

允许两个或更多启用使用位。停用使用位保留在配置中，但不参与报警判断。

### 本机配置

继续使用现有 `group-config.properties` 点位列表格式：每个点位独立保存角色、地码、名称、启用状态和顺序。

不增加版本字段，不迁移旧文件。原来只有一个使用位的配置加载和保存结果保持不变。

## 报警语义

对每个启用点位组计算：

```text
任一使用位无料 = 任一启用使用位处于无料或未查到状态
备用位不足 = 启用备用位有料数 < 配置的备用位最少有料数

规则成立 =
    点位组启用
    且报警规则启用
    且（未要求使用位条件，或任一使用位无料）
    且（未启用备用位数量条件，或备用位不足）
```

规则连续成立达到配置时间后触发报警。任何一次正常检测使规则不成立时，连续计时归零。

查询失败继续使用 `QUERY_FAILED`，不得把查询失败当作使用位无料；查询恢复后若仍异常，从零重新计时。

### 典型场景

| 使用位 1 | 使用位 2 | 备用位条件 | 使用位条件结果 | 最终规则 |
|---|---|---|---|---|
| 有料 | 有料 | 不足 | 不成立 | 不报警 |
| 无料 | 有料 | 不足 | 成立 | 达到持续时间后报警 |
| 有料 | 无料 | 不足 | 成立 | 达到持续时间后报警 |
| 无料 | 无料 | 充足 | 成立 | 不报警 |
| 停用且无料 | 有料 | 不足 | 不成立 | 不报警 |
| 查询失败 | 未获得 | 未获得 | 不计算 | 显示查询失败 |

表中最终规则假设“任一使用位无料”和“备用位不足”都参与报警判断。

## 界面行为

### 点位组管理

- “新增点位”继续允许把角色选择为“使用位”；
- 不再出现“只能有一个使用位”类限制；
- 保存和验证时使用中文提示“启用组至少需要一个启用的使用位”；
- 点位配置表可以连续出现多条使用位，每条都保留独立名称、地码和启用状态；
- 报警规则复选框文案统一为“任一启用使用位无料”。

### 状态看板与报警内容

- 每个使用位分别显示状态、地码和货码；
- 使用位区域按配置顺序排列，不合并成一个虚拟卡片；
- 汇总语句显示“使用位有料 X/Y；任一使用位无料：是/否”；
- 报警详情列出异常使用位名称，方便运维直接定位上料口；
- 备用位数量与状态展示保持现有语义。

## 数据流

```text
本机点位组配置
    -> 读取全部启用点位地码
    -> 现有只读批量查询
    -> PointStatusMapper 按点位逐条映射
    -> GroupMonitorLogic 聚合多个使用位和备用位
    -> GroupEvaluation
    -> 总览、状态看板、报警中心和本地日志
```

不增加数据库查询次数，不改变 SQL，不写数据库。

## 日志与兼容字段

现有日志中的 `usePointEmpty` 保留，含义明确为“任一启用使用位无料”。这样可以避免破坏旧日志读取和诊断流程。

点位详情继续逐条记录各使用位状态。日志不得包含密码、完整 JDBC URL、现场账号或栈信息。

## 错误处理

- 没有启用使用位：阻止保存，提示“启用组至少需要一个启用的使用位”。
- 没有启用备用位：阻止保存，使用直接中文提示。
- 地码重复：阻止保存并显示组名或组编号及重复地码。
- 某个使用位未查到：该使用位按无料参与规则，并在点位详情中明确显示“未查到”；只有整个查询失败时才进入 `QUERY_FAILED`。
- 配置读取损坏：沿用现有安全回退，不连接现场数据库。

## 测试策略

### 领域与配置

- 两个启用使用位可以创建、保存和重新加载；
- 零个启用使用位仍被拒绝；
- 两个使用位使用相同地码仍被拒绝；
- 单使用位旧配置保持兼容；
- 停用使用位不计入启用使用位数量和报警判断。

### 报警逻辑

- 两个使用位都有料时，使用位条件不成立；
- 任一使用位无料时，使用位条件成立；
- 任一使用位未查到时，使用位条件成立并保留具体原因；
- 停用使用位无料不触发使用位条件；
- 使用位条件成立但备用位充足时不报警；
- 使用位条件和备用位条件同时成立，达到持续时间后报警；
- 查询失败和恢复重新计时语义保持不变。

### 界面与回归

- 点位表可配置两条使用位；
- 状态看板同时显示两张使用位卡片且不重叠；
- 汇总和报警文本使用“任一使用位无料”，不出现 `USE`、`usePointEmpty` 等内部词；
- 完整 `build.ps1`、源码卫生、敏感信息、只读连接和本地 H2 预检全部通过。

## 验收标准

1. 一个启用点位组可保存两个或更多启用使用位；
2. 至少一个启用使用位仍为必填约束；
3. 任一启用使用位无料即可满足使用位异常条件；
4. 备用位不足和持续时间逻辑不变；
5. 停用使用位不参与报警；
6. 每个使用位在状态看板和详情中可单独识别；
7. 旧的单使用位配置无需迁移；
8. 数据库访问继续只读；
9. 全量测试与本地 H2 预检通过。

## 不在本阶段范围

- “全部使用位无料才报警”模式；
- 每个使用位单独配置持续时间；
- 使用位之间的优先级或主备切换；
- 数据库写入；
- CSV 点位组导入与导出。CSV 是后续独立子系统，复用本设计的多使用位模型。
