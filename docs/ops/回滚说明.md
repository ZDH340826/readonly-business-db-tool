# 回滚说明

适用版本：从 `0.5.0-rc.1` 回滚到现场已批准的旧版本。

## 原则

- 新旧版本必须并排解压，升级时不覆盖旧目录。
- 回滚只涉及本地程序、配置和日志，不修改、不迁移、不回滚业务数据库。
- 密码不在配置中，回滚后需要由现场人员重新输入。

## 升级前准备

1. 在“监控总览”点击“停止监控”，正常关闭窗口。
2. 记录旧版本号和目录，复制旧目录下 `data` 中的本地配置到受控备份位置。
3. 按现场制度保留必要的 `logs` 与 `diagnostics`，不要把它们提交到 Git。
4. 新版本解压到新目录，运行 `现场部署检查.bat` 和本地 H2 验收，再导入或重建现场配置。

## 触发回滚的建议条件

- 程序或预检无法启动，且在变更窗口内不能排除。
- 连接、查询失败语义、报警流程或点位组规则与已批准行为不一致。
- 日志/诊断无法满足现场审计，或出现敏感信息风险。
- 长时间试运行出现不可接受的资源、稳定性或兼容性问题。

## 回滚步骤

1. 在新版本点击“停止监控”，关闭程序并确认没有残留 Java 进程占用其目录。
2. 保留新版本目录作为问题现场；不要删除日志和诊断文件，按制度隔离敏感数据。
3. 打开旧版本目录，恢复与该旧版本匹配的 `data` 配置备份；不要把新旧格式文件盲目混用。
4. 运行旧版本自己的预检/自检（如有），确认目录和本地测试库可用。
5. 启动旧版本，重新输入密码，先测试连接，再执行一次受控“立即检测”。
6. 确认监控、报警和日志恢复后，再按变更流程结束回滚。

## 回滚后核对

- 记录新旧版本、回滚时间、原因、执行人和验证结果。
- 确认业务数据库没有执行任何结构或数据变更。
- 收集脱敏诊断包；真实配置、数据库文件和导出数据留在现场。
- 对 `0.5.0-rc.1` 的问题单独分析，未经重新构建、测试和批准不得覆盖旧版本。
