## 1. 本轮目标

- 点位组取消“只能有一个使用位”的限制，允许一个或多个启用使用位。
- 保留既有报警逻辑：任一启用使用位无料、状态异常、锁定或未查到时，使用位条件成立。
- 备用位下限、持续时间、查询失败独立语义和查询恢复重新计时保持不变。
- 面向非技术运维统一使用“地码”“货码”“使用位”“备用位”等中文词语。
- 只使用本地 H2，增加覆盖各种现场情况的公开虚构数据。
- 改善点位组配置与状态看板，不连接真实工作环境。

## 3. 报警语义

启用组必须至少有一个启用使用位和一个启用备用位。允许配置两个或更多使用位。

完整报警规则为：

```text
任一启用使用位无料或未查到
并且
启用备用位有料数量 < 设定下限
并且
以上状态按真实时间持续达到设定分钟数
```

补充边界：

- 所有使用位均有料时，使用位条件不成立。
- 备用位有料数量等于下限时，不算不足。
- 停用点位保留显示，但不参与数量计算。
- 查询失败为 `QUERY_FAILED`，不当作无料，不累计持续时间，不触发业务报警。
- 查询恢复后若仍异常，从零重新计时。

- 点位表新增“添加使用位”“添加备用位”两个明确入口。
- 角色下拉和表格内容只显示“使用位/备用位”，保存时转换为内部枚举。
- 点位表、数据查询和状态卡使用“地码/货码”。
- 状态摘要显示“使用位有料 X/Y”和“任一使用位无料：是/否”。
- 状态看板分为“使用位状态”和“备用位状态”，备用位按两列顺序排布。
- 状态卡同时显示别名、中文状态、地码、货码和原因；颜色只作辅助，文字可独立判断。

## 5. 本地 H2 场景

发布包包含 33 行公开虚构数据和 8 个点位组：

1. 双使用位中一个无料，备用位不足；
2. 双使用位全部有料，备用位不足；
3. 使用位无料，备用位数量充足；
4. 使用位状态异常、备用位锁定；
5. 配置中包含停用使用位和停用备用位；
6. 配置地码在数据库中未查到；
7. 使用位和备用位全部正常；
8. 备用位有料数量刚好等于下限。

`LocalDemoCatalog` 是数据库行和点位组定义的单一来源。`LocalTestDbTool reset` 可同时写本地数据库和组配置；`LocalTestDatabase` 会拒绝任何非 H2 配置。

## 7. 安全结论

- 本轮没有连接 PostgreSQL 或任何真实工作环境。
- 所有可变测试数据只写入临时目录或发布包内本地 H2。
- 生产查询仍为参数化固定 `SELECT` 和 JDBC 元数据读取。
- 未提交真实 IP、用户名、密码、私网地址、真实点位组、日志或现场配置。
- `docs/templates/点位组导入模板.csv` 保持为用户已有的未跟踪文件，本轮未修改、未暂存、未提交。
